# CloudSound 二次开发实录：从 YesPlayMusic 到双端适配音乐 Web 应用

今天把本地跑通、功能扩展、移动端适配、仓库整理和文档沉淀这一整套流程完整走了一遍。项目本体基于开源项目 [qier222/YesPlayMusic](https://github.com/qier222/YesPlayMusic) 继续开发，我们把它整理成了自己的 CloudSound，并围绕“更适合展示、答辩和持续开发”这个目标做了一轮较大改造。

## 一、项目背景

一开始我面对的不是“从零开始写一个播放器”，而是一个已经具备完整前端结构和播放器能力的开源项目。这个项目本质上是：

- 前端 Web 应用
- 本地或独立部署的后端 API 代理
- 浏览器端缓存与本地数据存储
- 通过第三方音乐接口获取数据

也就是说，它不是传统意义上必须自己从零设计数据库表、自己搭建完整后端业务逻辑的纯手搓项目，而是一个“前端播放器 + API 代理 + 本地存储”协作的架构。

这类项目很适合做二次开发，因为我们可以把精力集中到产品体验、功能创新和适配能力上。

## 二、最开始为什么跑不起来

项目最初试跑时，主要遇到了几类问题：

### 1. Node.js 版本不匹配

老项目依赖链对 Node 版本比较敏感。Node 过高时，部分旧依赖、构建链和脚本会出现兼容性问题。

### 2. 依赖未完整安装

播放器项目依赖较多，既有前端依赖，也有一些 Electron/Web 构建相关依赖。只要有关键依赖没装对，就会在 `serve` 或 `build` 时直接报错。

### 3. 本地 API 没接起来

这个项目页面能打开，不代表功能能完整工作。很多歌单、搜索、播放数据都要依赖 API 返回结果，所以“前端能跑”和“项目能正常使用”是两回事。

### 4. 开发环境和线上展示环境是两套逻辑

本地能访问 `localhost`，不代表手机局域网访问一定没问题。尤其是移动端访问开发服务器时，常常还会暴露出额外的编译错误、样式溢出和布局失衡问题。

## 三、这次具体做了什么

这轮开发不是只修一个点，而是连续推进了几个方向。

### 1. 跑通项目并整理部署流程

我们先把项目本地环境跑通，并整理出一份可复用的部署说明，避免以后再次踩环境坑。

沉淀文档包括：

- 本地运行与问题排查说明
- 开源归因文档
- 项目 README
- 今日技术博客

## 四、功能扩展

### 1. 缓存管理增强

原本缓存能力较弱，我们补充了更明确的缓存信息展示和清理能力，让“本地到底缓存了什么、清理后会影响什么”这件事更清晰。

这类改造的意义不只是功能增加，而是更接近一个真正可维护的播放器产品。

### 2. 本地播放历史与听歌统计

这是这次我认为很有价值的一项增强。

我们增加了：

- 本地播放历史
- 最近播放记录
- 听歌排行
- 本地统计维度沉淀

这一块的优势在于，不必依赖远端接口，也能形成用户自己的“使用画像”。对于个人项目、展示项目和毕设来说，这类功能很加分，因为它体现了“产品思考”而不只是“页面堆砌”。

### 3. 均衡器与音质入口整合

相比把均衡器放在设置页里，后面我们决定把它更贴近播放器本体，直接内嵌到播放器区域，让它更符合用户操作路径。

这是一个很重要的产品判断：

- 设置页适合放全局偏好
- 播放器本体适合放播放时高频使用的音频控制能力

所以这次调整不是简单挪位置，而是在优化交互结构。

### 4. PC / 手机 / iPad 双端适配

这一部分是今天后半段的重点。

最初手机访问时暴露出两个典型问题：

- 顶部导航在手机端没有适配，文字竖排挤压，观感很差
- 底部悬浮播放栏在移动端溢出，图标和文本布局失衡

后来进一步检查又发现：

- 首页卡片栅格在小屏上间距不协调
- 专辑详情页内容横向溢出
- 播放器区域在不同尺寸设备上缺少稳定的响应式布局

因此这次不是修一个局部样式，而是朝“双端适配”去改：

- PC 端保持原有视觉层次
- 手机端重排导航与播放器
- iPad 端保持介于桌面与手机之间的布局节奏

这也是后续还值得持续深入的一条主线。

## 五、今天这轮改造后的价值

改造完成后，CloudSound 相比原始基础项目，已经更偏向“可展示、可讲解、可持续扩展”的状态。

我认为这轮工作的几个核心价值是：

- 从“能打开”变成“能稳定运行”
- 从“开源项目原样体验”变成“有自己方向的二次开发版本”
- 从“桌面偏向界面”开始过渡到“多端可访问”
- 从“单纯播放器”增强为“具备本地数据沉淀能力的产品原型”

## 六、这次开发里最值得记住的经验

### 1. 不要把“页面能显示”当成项目跑通

真正跑通应该至少包含：

- 前端可访问
- API 可联通
- 核心功能可用
- 开发构建无致命错误
- 主要页面在目标设备上不崩

### 2. 二次开发项目最怕环境和分支混乱

如果不尽早整理：

- Node 版本会乱
- 依赖会乱
- 本地脚本会乱
- Git 分支也容易乱

这次连 GitHub 仓库展示异常，本质上也是分支默认展示不一致引发的错觉。

### 3. 功能设计要尽量贴近用户真实操作路径

均衡器如果藏得太深，用户几乎不会去用。把它贴近播放器主体，体验就会自然很多。

### 4. 移动端适配绝不是“缩小一下”

真正的移动端适配通常意味着：

- 信息层级重排
- 交互入口重组
- 控件尺寸调整
- 页面节奏重建

这也是为什么后续我们还会继续打磨。

## 九、结语

今天这轮工作最大的收获，不只是把项目“改出来了”，而是把它真正整理成了一个属于自己的开发版本。

从跑环境、补依赖、查 API、修页面，到扩展缓存、本地统计、均衡器，再到双端适配、仓库整理、文档输出，这些事情连起来，才构成一次完整的软件工程实践。

后面如果继续沿着产品化方向推进，CloudSound 完全可以从一个开源播放器二开项目，逐步变成一个更完整、更有个人特色的音乐产品原型。
